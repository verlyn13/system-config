# GitHub MCP Integration

Single source of truth for the GitHub MCP integration on this system.

For the general MCP framework (scopes, launch patterns, sync behavior),
see [`docs/mcp-config.md`](./mcp-config.md). For 1Password policy, see
[`docs/secrets.md`](./secrets.md).

This doc replaces the prior migration plan once the work it tracked
landed (commits `a8e0b64` through `fcf59fc`).

## Architecture

| Host | Auth | Config shape | Wrapper? |
|---|---|---|---|
| Claude Code CLI | PAT via stdio wrapper, currently hard-disabled | stdio entry pointing at `~/.local/bin/mcp-github-server` | **yes** |
| Cursor | PAT via stdio wrapper, currently hard-disabled | same stdio entry | **yes** |
| Codex CLI | PAT via stdio wrapper, currently hard-disabled | same stdio entry | **yes** |
| Windsurf | OAuth 2.1 + PKCE (native, shipped 1.12.41) | `serverUrl` only | no (verified working 2026-04-17) |
| Copilot CLI | Copilot-internal | **no synced entry** (built-in `github-mcp-server`) | n/a |

**Why the CLI hosts use the wrapper:** Claude Code's MCP SDK performs
OAuth discovery (RFC 9728 Protected Resource Metadata) before honoring
static Authorization headers. GitHub's remote MCP advertises OAuth
pointing at `github.com/login/oauth`, which does not support Dynamic
Client Registration. Claude Code's SDK errors out with "SDK auth failed:
Incompatible auth server: does not support dynamic client registration"
before falling back to our static bearer. The stdio wrapper sidesteps
this — the client sees a plain stdio MCP server, no OAuth discovery layer
to follow, and `mcp-remote` handles the HTTPS call with a pre-
authenticated bearer. Cursor uses the same shape. Codex also uses the
wrapper so bare launches do not need `GITHUB_PAT` in the process env.

Rendered shapes after `scripts/sync-mcp.sh`:

```json
// ~/.claude.json  (Claude Code)
"github": { "type": "stdio", "command": "/Users/verlyn13/.local/bin/mcp-github-server" }

// ~/.cursor/mcp.json  (Cursor)
"github": { "type": "stdio", "command": "/Users/verlyn13/.local/bin/mcp-github-server" }

// ~/.codeium/windsurf/mcp_config.json  (Windsurf)
"github": { "serverUrl": "https://api.githubcopilot.com/mcp/" }
```

```toml
# ~/.codex/config.toml  (Codex)
[mcp_servers.github]
command = "/Users/verlyn13/.local/bin/mcp-github-server"
```

All shapes are generated by `scripts/sync-mcp.sh`. Do not hand-edit these
entries; rerun sync instead. Hand-added sub-tables like
`[mcp_servers.github.tools.create_pull_request]` are supported and
survive regeneration if placed **outside** the `# BEGIN … # END` markers
in the Codex TOML.

The 1P item is dedicated to the MCP integration. Do not reuse
`op://Dev/github-dev-tools/token` (that entry serves `gh` CLI and other
general dev tooling). The non-secret credential record lives in
[`docs/secret-records.md`](./secret-records.md).

The 2026-05-08 bearer-token argv incident proved that `mcp-remote@0.1.x`
materializes the Authorization bearer into argv when launched with
`--header`. Do not relaunch GitHub MCP with a replacement PAT through the
current wrapper until the bridge architecture is fixed.

2026-05-08 rotation staging: the currently wired alias
`op://Dev/github-mcp/token` has been cleared and marked blocked. Paste the new
PAT value into the staging item
`op://Dev/github-jefahnierocks-macpro-mcp/credential` only. Do not wire that staging
alias into the wrapper until the replacement bridge no longer passes bearer
material through process argv.

2026-05-09 runtime containment: the deployed wrapper checks
`~/.local/state/system-config/mcp-github.disabled` before reading
`GITHUB_PAT` or calling `op`. While that file exists, all Claude Code, Cursor,
and Codex GitHub MCP launches exit `78`, even if a caller inherits a
`GITHUB_PAT` from the environment.

### Identity override: happy-patterns tree

When GitHub MCP is re-enabled after the no-argv bridge lands, any MCP host
launched from within `~/Organizations/happy-patterns/` can use the repo's
`.envrc` export of `GITHUB_PAT` from
`op://Dev/github-happy-patterns/token` (fine-grained PAT, resource owner
`happy-patterns-org`) before the host starts. The stdio wrapper at
`~/.local/bin/mcp-github-server` honors an inherited `GITHUB_PAT` and
skips its own `op read`, so the GitHub MCP server acts as the
`happy-patterns` identity for that session. Codex uses
the same wrapper and picks up the same override.
While the disable marker exists, this override path is intentionally blocked
too. After re-enable, launching from any other directory preserves the default
`jefahnierocks` identity via `op://Dev/github-mcp/token`.

Rotation procedure:
1. GitHub → Settings → Developer settings → Personal access tokens →
   Fine-grained tokens.
2. Create or rename the provider-side token to match the logical path:
   `github/jefahnierocks/macpro-mcp`. The token must be created under the
   `jefahnierocks` GitHub organization, not under a personal account.
3. Scope the token only to repos and org resources required by the
   Jefahnierocks MacPro MCP surface. Do not grant all `jefahnierocks` repos
   as a convenience.
4. During the current incident response, paste the new token into the staging
   1Password item `github-jefahnierocks-macpro-mcp`, field `credential`, from the
   1Password GUI. Do not paste it into the wired `github-mcp` alias until the
   no-argv bridge is in place.
5. Verify without printing the value:
   `op read --account my.1password.com "op://Dev/github-jefahnierocks-macpro-mcp/credential" >/dev/null && echo ok`
6. Verify the resolved token identity with `GH_TOKEN` in env, not argv.
7. If the rotation changes toolset scopes, update the PAT scope set
   below and the `X-MCP-Toolsets` header in `scripts/sync-mcp.sh` in
   lockstep, then rerun sync.

## Fine-grained PAT scopes

Target repository permissions (for the repos the MCP should see):

| Permission | Level |
|---|---|
| Metadata | Read |
| Contents | Read & write |
| Issues | Read & write |
| Pull requests | Read & write |
| Actions | Read & write |
| Workflows | Read & write |
| Commit statuses | Read |
| Deployments | Read |
| Discussions | Read & write |
| Code scanning alerts | Read & write |
| Secret scanning alerts | Read |
| Dependabot alerts | Read |
| Repository security advisories | Read & write |
| Administration | Read |

Account permissions: Profile (Read), Gists (Read & write), Starring
(Read).

Organization permissions: Metadata (Read), Members (Read), Administration
(Read), Custom properties (Read), Projects (Read & write).

**Not granted:** Secrets, Dependabot secrets, Environments, Pages,
Variables, Webhooks, Single file, Notifications.

**Scope filtering note:** fine-grained PATs do **not** get server-side
tool filtering (classic PATs do, as of Jan 28, 2026). The full GitHub
MCP tool surface would be exposed by default. The curated
`X-MCP-Toolsets` header below is the mechanism that enforces our
intended scope.

## Launch patterns

### Claude Code CLI, Codex CLI, and Cursor (stdio wrapper)

Do not expect authenticated GitHub MCP to connect while containment is active.
The stdio wrapper at `~/.local/bin/mcp-github-server` exits `78` when
`~/.local/state/system-config/mcp-github.disabled` exists, before reading
`op://Dev/github-mcp/token` or honoring an inherited `GITHUB_PAT`.

After the no-argv bridge is implemented and the disable marker is removed,
normal launches may resume. Launching via `op run --env-file=…` is still fine
and makes the wrapper fast-path by inheriting `GITHUB_PAT`, but the replacement
bridge must not materialize that value into process argv.

### Windsurf (GUI)

Launch normally. On first GitHub MCP interaction, Windsurf initiates
OAuth 2.1 + PKCE via GitHub's Protected Resource Metadata discovery
(RFC 9728). Consent once; Windsurf manages refresh tokens.

**Fallback:** if OAuth fails, the Windsurf version is below 1.12.41, or
refresh tokens stop working, revert Windsurf to the stdio wrapper by
changing `GITHUB_WINDSURF` in `scripts/sync-mcp.sh` to the same shape as
`GITHUB_CURSOR` and rerunning sync. One-commit revert.

### Copilot CLI

No configuration. The built-in `github-mcp-server` ships with the tool
and uses your Copilot credentials. Customize per-session:

```bash
copilot --enable-all-github-mcp-tools           # all toolsets
copilot --add-github-mcp-toolset <name>         # add a specific toolset
copilot --disable-builtin-mcps                  # turn it off
/mcp show github-mcp-server                     # from inside a session
```

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| GitHub wrapper: `disabled by ~/.local/state/system-config/mcp-github.disabled`, exit `78` | Intentional argv-exposure containment | Leave disabled until the no-argv bridge exists; do not bypass by pasting a PAT into the wired alias. |
| Claude Code / Cursor / Codex github status: failed, `SDK auth failed: Incompatible auth server: does not support dynamic client registration` | Config still uses native `type: "http"`; SDK is attempting OAuth DCR against GitHub | Rerun `scripts/sync-mcp.sh` — these hosts must use the stdio wrapper entry, not native HTTP. |
| GitHub wrapper: "unable to resolve GITHUB_PAT" | `op` can't reach the item; 1Password desktop integration off | Enable 1P Settings → Developer → Integrate with 1Password CLI; confirm `op vault get Dev --account my.1password.com` |
| Windsurf: stuck on connecting / no OAuth prompt | Windsurf version below 1.12.41 or OAuth discovery blocked | Upgrade Windsurf; if persistent, revert to stdio wrapper (see Fallback above) |
| Tool 403s at call time (e.g. `actions_run_trigger`) | PAT scope missing for that toolset | Add the missing Repository/Organization permission to the PAT; rotate via procedure above |
| Duplicate `[mcp_servers.*]` in Codex TOML | Sync ran against a Codex config without BEGIN/END markers | Wrap the existing `[mcp_servers.*]` block with `# BEGIN system-config managed MCP servers` / `# END …`, then rerun sync |
| Hand-added `[mcp_servers.github.tools.*]` disappeared after sync | Sub-table was inside the managed BEGIN/END block | Move the sub-table below the `# END` marker; TOML merges it into the parent table regardless of position |

## Rollback

Each architectural commit is independently reversible:

- Revert `fcf59fc` (per-host split) → returns to uniform wrapper across all three stdio hosts plus Codex bearer.
- Revert `754e45f` (wrapper uses mcp-remote) → returns to the prior stdio server; not recommended (deprecated npm package).
- Revert the entire stack → returns to pre-migration state.

After any revert, rerun `scripts/sync-mcp.sh` to propagate the rolled-back
shapes to live configs.
